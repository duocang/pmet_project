## 1. What lives here

PMET's longer-form documentation — anything too detailed to belong in
the repo-root README. Three flavours:

- **Method explanations** (`methods/`) — what PMET actually computes
  and why. Read these first if you're new.
- **Per-workflow audit docs** (`workflows/`) — for each of the four
  workflow scripts, a step-by-step walkthrough with a verification
  table that's regenerated by [tests/audit/](../tests/audit/). If
  the doc and the code disagree, the audit catches it.
- **Performance / archive** (`perf/`, `archive/`) — investigations
  with their results (e.g. why MinHash ships off by default) and
  pre-monorepo material kept for historical bisect.

[§2](#en-2) is a topic-to-document index; [§3](#en-3) is the directory
tree if you'd rather browse by file.

<a id="en-2"></a>

## 2. Where to start

If you came with a question, jump straight to the right doc:

- **"What does PMET do?"** → [methods/pmet.md](methods/pmet.md)
- **"How do I run a workflow?"** → repo-root [README.md](../README.md) §4
- **"Does this workflow do what its docs claim?"** → [workflows/](workflows/) — each doc walks every stage and PASS/FAIL-stamps the verification table
- **"How are the audit docs in workflows/ kept honest?"** → [tests/audit/README.md](../tests/audit/README.md) — they are regenerated by running each workflow against canonical inputs
- **"Why is the MinHash prefilter off by default?"** → [perf/minhash_calibration.md](perf/minhash_calibration.md)

## 4. Refreshing the workflow audit docs

The four `workflows/*.md` files are not pure prose — the verification
tables in them (file hashes, sizes, presence checks, PASS/FAIL
verdicts) come from actually re-running each workflow. After any
change to a workflow script:

```bash
make test-audit                                # regenerate all four
python3 tests/audit/generate.py promoter       # just one
```

Narrative parts stay; the tables get refreshed. If a check went
PASS → FAIL, either fix the workflow or update the doc to reflect the
intentional new behavior. See [tests/audit/README.md](../tests/audit/README.md)
for how the data flow works.

<a id="en-5"></a>

## 5. Conventions

- File names: lowercase + hyphens (`pmet.md`, `promoter-extraction.md`).
- Markdown only at the top of each subdir; figures live under `figures/`.
- One-language English when there is parity (`promoter-extraction.md`);
  Chinese duplicates moved to `archive/` or kept as separate `*-zh.md`
  when there's unique content (e.g. `methods/promoter-extraction-zh.md`).

## 4. 重跑 workflow 审计文档

`workflows/*.md` 那四份不是纯散文 —— 文档里的 verification 表（文件
hash、大小、存在性、PASS/FAIL）是真的把每个 workflow 跑一遍跑出来的。
改完 workflow 脚本后：

```bash
make test-audit                                # 重生成全部四份
python3 tests/audit/generate.py promoter       # 只重生成一份
```

叙事部分保留；表会刷新。某项 check 从 PASS 翻成 FAIL 的话，要么修
workflow、要么改文档承认是有意的新行为。数据流见
[tests/audit/README.md](../tests/audit/README.md)。

<a id="cn-5"></a>

## 5. 约定

- 文件名：小写 + 短横线（`pmet.md`、`promoter-extraction.md`）。
- 每个子目录最上面只放 markdown；图放 `figures/`。
- 英文有覆盖时只保英文（`promoter-extraction.md`）；汉文重复版搬到 `archive/`
  或独有内容时另起 `*-zh.md`（例如 `methods/promoter-extraction-zh.md`）。
